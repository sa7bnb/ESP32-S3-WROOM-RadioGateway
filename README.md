## What is this?

This is the design for a small, standalone **RoIP (Radio over IP) gateway** built around an **ESP32-S3** module. The idea is simple: breathe new life into an old analog FM radio by linking it to other radios over the internet — similar in spirit to existing RoIP solutions, but **without needing a computer connected to the radio**.

Most traditional RoIP / internet-linking setups (sound-card interfaces, AllStarLink nodes, EchoLink, etc.) require a PC, Raspberry Pi, or similar computer wired between the radio and the network. This project aims to collapse all of that into a single small board that connects directly to the radio's data port on one side and to WiFi on the other.

The board talks to the radio through its packet/DATA port: it pulls receive audio out, feeds transmit audio in, keys PTT, and senses the COR/squelch line. It then connects over WiFi to a server (self-hosted) that ties multiple radios together across the internet — letting otherwise isolated analog radios talk to each other over distance.

The reference target for this design is the **Yaesu FT-7900R** (and the closely related FT-78xx / FT-88xx / FT-89xx family, which share the same 6-pin mini-DIN DATA jack and pinout), but the concept applies to any analog FM radio with a comparable data/packet port.

## Goals

- **No PC required at the radio** — the gateway is self-contained.
- **Direct connection** to the radio's 6-pin mini-DIN DATA port.
- **Isolated** audio and control lines to keep ground loops and RF/digital noise out of the audio path.
- **Small, low cost, few components** — built around a single ESP32-S3 module with mostly passive supporting parts.
- **Self-hosted linking** — connects to a server you run, in the same family of ideas as other RoIP systems.

## Repository contents

All files needed to **manufacture, develop, and test** the PCB are available here in this GitHub repository — including the schematic, PCB design files, and related design resources. As the project progresses, firmware and server-side code will be added here as well.

## Non-goals (at least for now)

- This is not a packet/TNC modem; it is intended for **voice** linking.
- It does not (yet) aim for compatibility with any specific existing linking protocol — the link protocol is part of the ongoing work.

## Hardware overview

The board is built around an **ESP32-S3-WROOM-1 (N16R8)** module — 16 MB flash, 8 MB PSRAM, integrated PCB antenna. The S3 was chosen over smaller parts because two-way real-time audio plus a network connection benefits from the dual cores, PSRAM (for jitter buffering), and headroom for optional audio compression.

Power comes in over **USB-C** (also used for programming and debugging via the S3's built-in USB-Serial-JTAG), regulated down to 3.3 V on-board.

### Radio interface (FT-7900R DATA jack, 6-pin mini-DIN)

| Pin | Label | Direction | Used for |
|-----|-------|-----------|----------|
| 1 | PKD (DATA IN) | Gateway → Radio | Transmit audio into the radio's modulator |
| 2 | GND | — | Radio-side signal ground (RADIO-GND) |
| 3 | PTT | Gateway → Radio | Key the transmitter (ground to transmit) |
| 4 | RX9600 | Radio → Gateway | (not used — flat discriminator audio) |
| 5 | RX1200 | Radio → Gateway | Receive audio (de-emphasized, processed) |
| 6 | PKS (SQL) | Radio → Gateway | COR / squelch status (+5 V open, 0 V closed) |

Per the FT-7900R manual the relevant levels are: RX1200 output ≤ 300 mV p-p, PKD input ≤ 40 mV p-p (1200 bps mode), and squelch line +5 V (open) / 0 V (closed). These values drive the attenuation and level-setting choices in the design.

### Signal chains

**Receive audio (RX1200 → ESP32 ADC):**
Audio is taken from the radio, passed through an isolation transformer, attenuated with a fixed + trimmer divider, AC-coupled, biased to mid-rail (~1.65 V), low-pass filtered, and fed into an ADC1 channel on the ESP32-S3. A trimmer allows the receive level to be matched to the radio.

**Transmit audio (ESP32 PWM → PKD):**
Audio is generated by the ESP32 (PWM / sigma-delta), passed through a two-stage RC low-pass filter to remove the PWM carrier, attenuated heavily (the PKD input wants only tens of millivolts), level-set with a trimmer, and sent through an isolation transformer to the radio's data input.

**PTT (ESP32 → radio):**
Driven through an optocoupler so the keying line is galvanically isolated. An open-collector-style output grounds the radio's PTT line to transmit. A pulldown keeps the transmitter from being keyed accidentally during boot.

**COR / squelch (radio → ESP32):**
Sensed through a second optocoupler. The radio's +5 V squelch line drives the optocoupler's LED; the ESP32 reads the transistor side. Logic is inverted in firmware, and polarity is intended to be configurable since this varies between radios.

### Galvanic isolation

A key design choice is that **all four signals crossing between the radio and the gateway are isolated**: two audio transformers (RX and TX) and two optocouplers (PTT and COR). The board is split into two ground domains — **RADIO-GND** (radio side) and the board's own **GND** — which must never meet except through the isolation components. This is intended to keep mains hum, WiFi noise, and PWM switching noise out of the audio. Because the radio side carries no active electronics that need powering, no isolated DC/DC supply is required; RADIO-GND exists purely as an isolated reference.

> Note: full isolation depends on correct PCB layout — RADIO-GND and GND must be drawn as two physically separate copper areas with only the transformers and optocouplers spanning the gap.

## Status & roadmap

**Current status:** schematic design only. Not built, not tested, no firmware.

Things that still need to happen before this is usable:

- [ ] Finish and review the full schematic
- [ ] PCB layout (with careful attention to the two isolated ground domains and antenna keep-out)
- [ ] Build a first prototype
- [ ] Verify power, boot, and USB programming
- [ ] Bring up and test each signal chain (RX audio, TX audio, PTT, COR) against a real FT-7900R
- [ ] Trim audio levels against the radio's actual RX1200 / PKD levels
- [ ] Write the gateway firmware (audio I/O, WiFi, link protocol, PTT/COR handling)
- [ ] Build / integrate the server side for linking radios over the internet
- [ ] Field testing

Component values and pin assignments in the current design are starting points and are expected to change during bring-up.
